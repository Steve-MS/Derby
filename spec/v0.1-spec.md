## 1. Deliverable Shape

**What:** A Python CLI tool under `C:\Users\stevenn\race-analysis\` with five subcommands:

| Command | Purpose |
|---------|---------|
| `fetch` | Scrape/ingest racecard data from public web sources into canonical JSON |
| `score` | Apply the composite scoring model to each runner in a race |
| `predict` | Rank runners per race, output predictions |
| `backtest` | Compare predictions against actual results, compute hit-rate metrics |
| `report` | Generate markdown reports per race (human-readable) |

**Why Python CLI:**
- Steve is comfortable with Python — low friction to hack/extend.
- No deployment infra needed — runs local on his machine.
- Libraries available: `requests`, `beautifulsoup4`, `rich` (for CLI output), `pytest`.
- No web UI, no server, no docker. Just `python -m race_analysis <cmd>`.

**Entry point:** `python -m race_analysis` (package invocation) or `python cli.py <cmd>`.

**Dependencies:** Python 3.11+, `requests`, `beautifulsoup4`, `rich`, `pytest` (dev only). All pip-installable, no paid APIs.

## 4. Methodology — Composite Scoring Model

### 4.1 Signals

| Signal | Description | Weight | Normalization |
|--------|-------------|--------|---------------|
| Official Rating (OR) | BHA handicap rating | 0.20 | Linear scale 0–1 across field |
| Racing Post Rating (RPR) | RP's private rating | 0.15 | Linear scale 0–1 across field |
| Topspeed (TS) | Speed figure | 0.10 | Linear scale 0–1 across field |
| Recent Form | Weighted avg of last-5 finishes (most recent = heaviest) | 0.15 | Decay weights: [0.35, 0.25, 0.20, 0.12, 0.08] |
| Trainer 14-day strike rate | % winners in last 14 days | 0.08 | Raw percentage |
| Jockey course strike rate | Jockey's win % at this specific course | 0.07 | Raw percentage |
| Going suitability | Does horse's form on this going type match? | 0.08 | Binary/graded: 1.0 (proven), 0.7 (likely), 0.4 (untested), 0.1 (poor) |
| Course-and-distance (C&D) | Wins/places at this course AND distance | 0.05 | Ratio of wins+places to runs, capped at 1.0 |
| Draw bias | Stall advantage for sprint races (Epsom 5f) | 0.04 | Course-specific lookup table; 0 for races ≥1m |
| Class change | Moving up/down in class | 0.04 | +0.1 per class drop, -0.1 per class rise, clamped [-0.3, +0.3] |
| Days since last run | Freshness/fitness indicator | 0.04 | Bell curve: peak at 14–28 days, penalize <7 or >90 |

**Total weights: 1.00**

### 4.2 Composite Score Formula

```
score(runner) = Σ (weight_i × normalized_signal_i)
```

Each signal is normalized to [0, 1] range within the race field. Final scores rank runners from highest (best chance) to lowest.

### 4.3 Tuning

Initial weights above are educated guesses. Kaylee owns weight tuning post-backtest. If Ladies Day backtest shows systematic bias (e.g., form overpowered by class), she adjusts weights for Derby Day run.

## 5. Pipeline

```
fetch → normalize → score → rank → output
```

### 5.1 Fetch

```bash
python -m race_analysis fetch --date 2026-06-06 --venue epsom
```

Downloads racecards into `data/raw/epsom-2026-06-06/`. One HTML file per race.

### 5.2 Normalize

Runs automatically after fetch (or standalone):
```bash
python -m race_analysis fetch --date 2026-06-06 --venue epsom --normalize
```

Outputs canonical JSON to `data/cards/epsom-2026-06-06/race-{time}.json`.

### 5.3 Score

```bash
python -m race_analysis score --date 2026-06-06
```

Applies composite model. Outputs scored runners to `outputs/predictions/epsom-2026-06-06/race-{time}-scores.json`.

### 5.4 Predict

```bash
python -m race_analysis predict --date 2026-06-06
```

Ranks runners per race. Outputs:
- `outputs/predictions/epsom-2026-06-06/race-{time}-prediction.json` (top-3 picks + full ranking)
- Summary to stdout.

### 5.5 Report

```bash
python -m race_analysis report --date 2026-06-06
```

Generates `outputs/reports/epsom-2026-06-06.md` — a markdown report with:
- Per-race: top 3 picks, scores, rationale (which signals dominated)
- Summary table across the day
- Confidence indicators (margin between #1 and #2)

### 5.6 Backtest

```bash
python -m race_analysis backtest --date 2026-06-05
```

Requires:
1. Predictions already generated for that date
2. Actual results ingested: `data/results/epsom-2026-06-05/race-{time}-result.json`

Outputs hit-rate metrics (see §6).

## 6. Backtest Contract

### 6.1 Purpose

Ladies Day (Friday 5 June 2026) serves as validation. We run the full pipeline on Friday's card, wait for actual results, then measure prediction quality.

### 6.2 Metrics

| Metric | Definition | Trust Threshold |
|--------|-----------|-----------------|
| Winner Hit Rate | % of races where our #1 pick wins | ≥ 1 winner out of 7 races (≈14%) |
| Top-3 Hit Rate | % of races where our #1 pick finishes in the top 3 | ≥ 50% of races |
| Place Accuracy | % of races where ≥2 of our top-3 picks finish top-5 | ≥ 40% of races |
| Rank Correlation | Spearman ρ between predicted rank and actual finish position | ρ > 0.3 (weak positive) |

### 6.3 Trust Decision

**Model is TRUSTED for Derby Day if ANY of these hold:**
1. Winner Hit Rate ≥ 14% (at least 1 winner from ~7 races), OR
2. Top-3 Hit Rate ≥ 50%, OR
3. Place Accuracy ≥ 40% AND Rank Correlation > 0.3

**Model is DISTRUSTED if:**
- Winner Hit Rate = 0% AND Top-3 Hit Rate < 30%

If distrusted, Kaylee retunes weights before Saturday's run. If tuning doesn't help, we run Saturday with a "low confidence" flag on the report.

### 6.4 Jayne's Test Harness Contract

Jayne owns the backtest implementation. Her contract:

**Input:**
- `outputs/predictions/{venue}-{date}/race-{time}-prediction.json` — our predictions
- `data/results/{venue}-{date}/race-{time}-result.json` — actual finishing positions

**Result schema:**
```json
{
  "date": "2026-06-05",
  "venue": "Epsom",
  "races_analysed": 7,
  "winner_hit_rate": 0.14,
  "top3_hit_rate": 0.57,
  "place_accuracy": 0.43,
  "rank_correlation": 0.35,
  "trusted": true,
  "per_race": [
    {
      "race_time": "14:00",
      "predicted_winner": "Horse A",
      "actual_winner": "Horse B",
      "predicted_top3": ["Horse A", "Horse C", "Horse D"],
      "actual_top3": ["Horse B", "Horse A", "Horse C"],
      "our_pick_finished": 2,
      "hit": false,
      "top3_hit": true
    }
  ]
}
```

**Output:** `outputs/backtest/epsom-2026-06-05-backtest.json` + stdout summary.

## 7. Out of Scope (v0.1)

| NOT doing | Reason |
|-----------|--------|
| Live odds scraping | Complexity + ToS risk. Use morning prices only (manually noted or from racecard). |
| Betting integration | No placing bets programmatically. This is analysis, not a bot. |
| In-running adjustments | No live updates. Predictions are pre-race only. |
| Machine learning models | Overkill for v0.1. Start with weighted heuristics; ML is a future v0.2 consideration. |
| Multiple venues | v0.1 targets Epsom only. Structure allows extension but we don't build it now. |
| Historical database | No multi-year form database. Use current season racecard data only. |
| Odds-based probability models | No Kelly criterion, no implied probability from markets. |
